## A. `num_layers` is structural, not just an iteration count

Covered by `test_num_layers_instantiates_separate_lstms[L]` for `L ∈ {1, 2, 3, 4, 6}` and the slope test `test_lstm_block_param_count_scales_with_num_layers`. For each `L`, we instantiate `RSNN_LSTM_Reg(pe_in_dim=8, pe_out_dim=16, hid_dim=64, out_dim=1, num_layers=L, n_emb=6, reduce='sum')` and assert (i) `len(model.rnn_layers) == L`, (ii) every entry is an `nn.LSTM` with `bidirectional=True` and `num_layers=1` (so the depth comes from the outer Python loop, not from a stacked PyTorch LSTM), and (iii) the `weight_hh_l0` / `weight_ih_l0` tensors of consecutive layers are *distinct Python objects* (no aliasing). The parameter-count test confirms `sum(p.numel() for p in rnn_layers.parameters())` is strictly increasing in `L` and that the per-added-layer cost is constant to within 5 % — i.e. the model truly allocates one extra `nn.LSTM(2*hid, hid, bidirectional=True)` per additional layer. **Result: passes for every `L`.**

## B. `forward()` actually uses every LSTM layer

Covered by `test_forward_invokes_every_lstm_layer_exactly_once[L]` for `L ∈ {1, 2, 3, 4}`. We build a tiny synthetic `Batch` of two graphs whose `walk_emb`/`walk_ids`/`walk_pe`/`lengths` shapes match those produced by `sample_walks_adaptive` (`walk_emb`: `(NW, MAX_LEN)`, `walk_ids`: `(1, NW, MAX_LEN)`, `walk_pe`: `(NW, MAX_LEN, pe_in)`, `lengths`: `(NW,)`), register a forward hook on each `model.rnn_layers[i]`, run a single forward pass, and assert that every per-layer counter equals exactly 1. This rules out two failure modes simultaneously: (a) a layer being silently skipped (counter == 0) and (b) the same layer being re-invoked in place of another (counter > 1 on one, == 0 on another). The forward output shape `(B=2, out_dim=1)` is also checked. **Result: passes for `L = 1, 2, 3, 4`.**

## C. Permutation properties

Covered by `test_output_is_invariant_to_walk_permutation[reduce]` for `reduce ∈ {sum, mean}` and `test_output_is_sensitive_to_timestep_permutation_within_walk`. We compute `y1 = model(batch)` once, then build a permuted batch where the *walks* of each graph are reordered along the `NW` axis (with non-trivial permutations `(2, 0, 1)` and `(1, 2, 0)`) and check `‖y1 - y2‖_∞ < 1e-4`. This confirms that — under sum/mean graph readout — the model is invariant to walk ordering even though each walk is processed by a sequence model: `graph_ns` depends only on the set of node ids, the scatter-mean over `walk_ids_proc_flat_masked` is a set-aggregator, and the final scatter sum/mean over per-graph nodes is a set-aggregator. The third sub-test reverses every walk's timesteps (`torch.flip(..., dims=[1/2])`) and asserts `‖y1 - y3‖_∞ > 1e-4`: the bidirectional LSTM is *not* time-symmetric (reversing a sequence permutes which positions get associated with the forward/backward hidden states), so the per-step LSTM features the scatter-mean averages over differ, and the readout responds accordingly. This is the documented behaviour, not a bug. **Result: passes — invariance holds under walk permutation, sensitivity holds under within-walk timestep reversal.**
